# 实施任务：交易删除保护 + 文档 Line Items 交易状态标注

## Part A: 交易删除保护

### Task 1: 后端 — 新增删除预检查端点
- [x] 在 `backend/app/api/v1/endpoints/transactions.py` 新增 `GET /transactions/{id}/delete-check` 端点
- [x] 查询交易的 `document_id` 和 `source_recurring_id`
- [x] 如果有 `document_id`，查询该文档关联的交易总数（`SELECT COUNT(*) FROM transactions WHERE document_id = X`）并获取文档名称
- [x] 返回结构：`{ can_delete: bool, warning_type: "document_only" | "document_multi" | "recurring" | null, document_id: int?, document_name: str?, linked_transaction_count: int?, is_from_recurring: bool }`
- [x] `warning_type` 逻辑：`document_only`（绑定文档且是唯一交易 → 禁止删除）、`document_multi`（绑定文档但还有其他交易 → 需确认）、`recurring`（来自定期交易 → 需确认）、`null`（无关联 → 可直接删除）

**涉及文件**: `backend/app/api/v1/endpoints/transactions.py`
**对应需求**: 2.1, 2.2, 2.3

### Task 2: 后端 — 修改单笔删除端点支持 force 参数
- [x] 修改 `DELETE /transactions/{id}` 端点，新增 query 参数 `force: bool = False`
- [x] 删除前执行与 Task 1 相同的关联检查逻辑
- [x] 如果 `warning_type == "document_only"`，无论 force 值如何，返回 HTTP 409 Conflict，附带提示信息
- [x] 如果 `warning_type` 为 `"document_multi"` 或 `"recurring"` 且 `force=False`，返回 HTTP 409 Conflict，附带 warning 信息和 `requires_confirmation: true`
- [x] 如果 `warning_type` 为 `"document_multi"` 或 `"recurring"` 且 `force=True`，执行删除
- [x] 如果无关联（`warning_type == null`），直接删除（保持现有行为不变）

**涉及文件**: `backend/app/api/v1/endpoints/transactions.py`
**对应需求**: 2.1, 2.2, 2.3, 3.1

### Task 3: 后端 — 修改批量删除端点支持预检查
- [x] 修改 `POST /transactions/batch-delete` 端点，新增 body 参数 `force: bool = False`
- [x] 遍历所有待删除交易，执行关联检查
- [x] 如果 `force=False` 且存在有关联的交易，返回预检查结果：`{ blocked: [{id, reason, document_name}], needs_confirmation: [{id, warning_type, document_name, linked_count}], safe: [id] }`，不执行删除
- [x] 如果 `force=True`，删除所有 `safe` + `needs_confirmation` 的交易（跳过 `blocked`），返回 `{ deleted: [id], blocked: [{id, reason}], count: int }`
- [x] 无关联交易的批量删除行为保持不变

**涉及文件**: `backend/app/api/v1/endpoints/transactions.py`
**对应需求**: 2.4, 3.2

### Task 4: 前端 — transactionService 新增 deleteCheck 方法
- [x] 在 `frontend/src/services/transactionService.ts` 新增 `deleteCheck(id: number)` 方法，调用 `GET /transactions/{id}/delete-check`
- [x] 修改 `delete(id: number, force?: boolean)` 方法，支持传递 `force` query 参数
- [x] 修改 `batchDelete(ids: number[], force?: boolean)` 方法，支持传递 `force` body 参数

**涉及文件**: `frontend/src/services/transactionService.ts`
**对应需求**: 2.1, 2.2, 2.3, 2.4

### Task 5: 前端 — TransactionsPage 删除确认弹窗逻辑
- [x] 修改 `handleDeleteTransaction`：删除前先调 `transactionService.deleteCheck(id)`
- [x] 根据 `warning_type` 显示不同确认弹窗：
  - `null` → 简单确认"确定删除？"后调 `delete(id)`
  - `recurring` → 显示"这笔来自定期交易，删除后下次仍会自动生成，确定删除？"，确认后调 `delete(id, true)`
  - `document_multi` → 显示"此交易来自文档 {name}（共 N 笔），仅删除这一笔，确定？"，确认后调 `delete(id, true)`
  - `document_only` → 显示"此交易关联了文档 {name}，请从文档管理中修改"，禁用删除按钮
- [x] 修改 `handleBatchDelete`：先调 `batchDelete(ids, false)` 获取预检查结果，显示摘要弹窗（N 笔可直接删除、M 笔需确认、K 笔被阻止），用户确认后调 `batchDelete(ids, true)`

**涉及文件**: `frontend/src/pages/TransactionsPage.tsx`
**对应需求**: 2.1, 2.2, 2.3, 2.4
